## Package boundaries

| Package | Responsibility |
| --- | --- |
| `packages/client` | Gardener-facing PWA UI |
| `packages/admin` | Operator-facing dashboard UI |
| `packages/shared` | Hooks, providers, stores, business logic |
| `packages/contracts` | Solidity modules + deployment scripts |
| `packages/indexer` | Envio event indexing and GraphQL surface |
| `packages/agent` | Automation/bot workflows |

## Architecture principles

- DRY for shared hooks/query keys/deployment addresses.
- KISS for chain/environment model.
- YAGNI for config surface and feature toggles.
- Separation of concerns across packages.

## Contract module system

GardenToken uses a **hub-and-spoke pattern** with typed module slots. Each module is a UUPS-upgradeable proxy deployed via CREATE2. Optional modules degrade gracefully — every call is wrapped in `if (address != 0)` + `try/catch`.

```mermaid
graph LR
    GT["GardenToken<br/><small>Hub</small>"]

    GT --- |"REQUIRED"| HM["HatsModule<br/><small>RBAC via Hats Protocol</small>"]
    GT --- |"REQUIRED"| CT["communityToken<br/><small>Shared ERC-20</small>"]
    GT -.- |"optional"| KG["KarmaGAPModule<br/><small>Impact attestations</small>"]
    GT -.- |"optional"| OCT["OctantModule<br/><small>Yield vaults</small>"]
    GT -.- |"optional"| GM["GardensModule<br/><small>V2 communities</small>"]
    GT -.- |"optional"| CJ["CookieJarModule<br/><small>Petty cash jars</small>"]
    GT -.- |"optional"| AR["ActionRegistry<br/><small>Action domain config</small>"]
    GT -.- |"optional"| ENS["GreenGoodsENS<br/><small>ENS subdomains via CCIP</small>"]

    style GT fill:#fff9c4,stroke:#f9a825
```

## Contract-to-indexer event mapping

The Envio indexer watches specific contract events and materializes them into GraphQL entities. EAS attestations (works, approvals, assessments) are **not** re-indexed — they're queried directly from easscan.org.

```mermaid
graph LR
    subgraph Contracts["Contract Events"]
        E1["GardenToken<br/><code>GardenMinted</code>"]
        E2["HatsModule<br/><code>RoleGranted</code><br/><code>RoleRevoked</code>"]
        E3["ActionRegistry<br/><code>ActionRegistered</code><br/><code>GardenDomainsUpdated</code>"]
        E4["GardenAccount<br/><code>NameUpdated</code><br/><code>DescriptionUpdated</code><br/><code>…</code>"]
        E5["OctantModule<br/><code>VaultCreated</code><br/><code>HarvestTriggered</code><br/><code>DonationAddressUpdated</code><br/><code>EmergencyPaused</code>"]
        E6["OctantVault<br/><code>Deposit</code><br/><code>Withdraw</code>"]
        E7["YieldSplitter<br/><code>YieldSplit</code>"]
        E8["HypercertMinter<br/><code>TransferSingle</code><br/><code>ClaimStored</code>"]
    end

    subgraph Entities["Indexed Entities"]
        G["Garden"]
        Gd["Gardener"]
        A["Action"]
        DD["GardenDomains"]
        GV["GardenVault"]
        VD["VaultDeposit"]
        VE["VaultEvent"]
        YA["YieldAllocation"]
        HC["Hypercert"]
        HCC["HypercertClaim"]
    end

    E1 --> G
    E2 --> G
    E2 --> Gd
    E3 --> A
    E3 --> DD
    E4 --> G
    E5 --> GV
    E6 --> VD
    E6 --> VE
    E6 --> GV
    E7 --> YA
    E8 --> HC
    E8 --> HCC

    style Contracts fill:#fce4ec,stroke:#c62828
    style Entities fill:#e3f2fd,stroke:#1565c0
```

**Dynamic contract registration**: When a garden is minted, its TBA address is registered for `GardenAccount` events. When a vault is created, its address is registered for `Deposit`/`Withdraw` events.

**Indexer boundary**: The Envio indexer only covers core Green Goods state. Several modules have stub handler files that are explicitly externalized: cookie jars, ENS lifecycle, Gardens V2 communities/pools, marketplace orders, and power registry audits are all read via direct RPC calls or external subgraphs at query time. EAS attestations (works, approvals, assessments) are queried from easscan.org, not re-indexed.

## Offline-first job queue lifecycle

The client PWA supports offline work submission via an IndexedDB-backed job queue. Jobs are queued instantly, then flushed when connectivity returns.

```mermaid
sequenceDiagram
    actor User as Gardener (PWA)
    participant UI as React UI
    participant JQ as Job Queue (IndexedDB)
    participant SW as Service Worker
    participant IPFS as IPFS (Storacha)
    participant Chain as EAS Contract

    User->>UI: Submit work (photos + details)
    UI->>JQ: submitWorkToQueue()
    JQ-->>UI: 0xoffline_... (instant placeholder)
    UI-->>User: "Submitted" (optimistic)

    Note over JQ,SW: When online + smart account ready

    alt Online immediately
        JQ->>JQ: processJob()
        JQ->>IPFS: Upload media
        IPFS-->>JQ: CIDs
        JQ->>Chain: EAS.attest() via UserOperation
        Chain-->>JQ: tx receipt
        JQ-->>UI: job:completed event
        UI->>UI: Invalidate query cache
    else Comes online later
        SW->>JQ: BACKGROUND_SYNC event
        JQ->>JQ: flush() — process all pending
        loop For each queued job
            JQ->>IPFS: Upload media
            JQ->>Chain: EAS.attest()
        end
        JQ-->>UI: job:completed events
    end

    Note over UI: 2s delayed invalidation for indexer lag
```

**Retry policy**: exponential backoff (1s base, doubling, max 60s), max 5 retries before permanent failure. Mutex prevents concurrent flush operations.

## Garden minting sequence

When an operator mints a garden, GardenToken fans out to all configured modules in two phases.

```mermaid
sequenceDiagram
    actor Op as Operator
    participant GT as GardenToken
    participant TBA as GardenAccount (TBA)
    participant HM as HatsModule
    participant KG as KarmaGAPModule
    participant OCT as OctantModule
    participant GM as GardensModule
    participant CJ as CookieJarModule
    participant ENS as GreenGoodsENS

    Op->>GT: mintGarden(name, desc, location, …)
    GT->>GT: _safeMint(tokenId)
    GT->>TBA: Deploy tokenbound account (ERC-6551)

    rect rgb(232, 245, 233)
        Note over GT,GM: Phase 1 — Role & Governance
        GT->>HM: createGardenHatTree(garden, roles)
        HM-->>HM: Create 6-role hat tree
        GT->>KG: createProject(garden) [try/catch]
        GT->>OCT: onGardenMinted(garden) [try/catch]
        Note over OCT: Creates vault per supported asset
        GT->>GM: onGardenMinted(garden) [try/catch]
        Note over GM: Creates community + signal pools
    end

    rect rgb(255, 243, 224)
        Note over GT,ENS: Phase 2 — Integrations
        GT->>CJ: onGardenMinted(garden) [try/catch]
        Note over CJ: Deploys per-asset cookie jars
        GT->>ENS: registerGarden(slug) [try/catch]
        Note over ENS: CCIP message to L1
    end

    GT->>TBA: initialize(name, desc, location, …)
    GT-->>Op: GardenMinted event
```

## Work submission and approval

The core user flow: a gardener submits work, an operator reviews it, and the attestation chain is anchored on EAS.

```mermaid
sequenceDiagram
    actor G as Gardener
    actor Op as Operator
    participant EAS as EAS Contract
    participant WR as WorkResolver
    participant WAR as WorkApprovalResolver
    participant HM as HatsModule
    participant AR as ActionRegistry
    participant KG as KarmaGAPModule

    G->>EAS: attest(WorkSchema)
    EAS->>WR: onAttest() validation
    WR->>HM: isGardener(attester)?
    HM-->>WR: true
    WR->>AR: getAction(actionUID) valid + active?
    AR-->>WR: true
    WR-->>EAS: allow
    EAS-->>G: workUID

    Note over G,Op: Later — operator reviews

    Op->>EAS: attest(WorkApprovalSchema)
    EAS->>WAR: onAttest() validation
    WAR->>WAR: Verify workUID matches garden
    WAR->>HM: isOperator(attester)?
    HM-->>WAR: true
    WAR->>WAR: Validate confidence + method

    alt approved == true
        WAR->>KG: createImpact() [try/catch]
        KG-->>WAR: impact attestation
    end

    WAR-->>EAS: allow
    EAS-->>Op: approvalUID
```
