## Abstract

_Can you explain the whole project and its expected outcome(s)_

Evidence Commons is an open-source toolkit that lets any web application capture field evidence offline, persist it locally, and submit it as a verifiable on-chain attestation when connectivity returns.

The toolkit extracts battle-tested infrastructure from the Green Goods protocol — a regenerative impact documentation platform live in 8+ communities across Nigeria, Brazil, South Africa, and Uganda — into reusable, framework-agnostic libraries:

**1. Offline Evidence Queue** — An IndexedDB-backed job queue with draft persistence, media file serialization, exponential backoff retry, and background sync via Service Workers. Currently ~2,100 lines of production code handling real submissions from field workers on mid-range Android devices with intermittent 2G/3G connectivity.

**2. Impact Attestation Standard** — A formal specification for multi-stage evidence verification using the Ethereum Attestation Service (EAS). Defines the Work-Approval-Assessment attestation chain, resolver validation rules, role-based access requirements, and the Common Impact Data Standard (CIDS) Activity-Output-Outcome-Impact framework. Published as both a specification document and an npm encoder/decoder library.

**3. Passkey Authentication Bridge** — A reusable WebAuthn-to-ERC-4337 smart account pipeline that takes users from biometric scan to first on-chain action with zero wallet setup. Abstracts bundler and paymaster logic behind configurable provider interfaces so any project can offer "zero-wallet onboarding" without vendor lock-in.

**4. Reference Implementation** — A complete example application demonstrating all three libraries working together, with end-to-end tests and comprehensive documentation.

These components exist today as application-specific code inside Green Goods. Funding is needed to extract them into generic, well-tested, well-documented libraries that any impact verification, citizen science, or community reporting project can adopt. The individual building blocks (IndexedDB, EAS, WebAuthn) exist separately, but integrating them into a reliable offline-to-attestation flow for low-connectivity environments is hard engineering that Green Goods has already solved in production.

All outputs will be MIT-licensed npm packages, specification documents, and reference documentation.

### Comparison

_Compare your own project with existing or historical efforts._

Evidence Commons occupies a unique position — no existing project publishes an open-source, offline-first evidence-to-attestation pipeline as reusable libraries:

- **Ceramic / ComposeDB** — General-purpose decentralized data infrastructure. Provides document storage and querying but offers no offline-first primitives, no attestation chain model, and no passkey authentication integration. Evidence Commons focuses specifically on the evidence capture-to-verification workflow.

- **IPFS tooling (Protocol Labs)** — Strong at content addressing and distributed storage, but provides no opinion on evidence workflows, multi-stage verification chains, or offline job queuing. Evidence Commons uses IPFS for media storage but adds the entire verification and attestation layer on top.

- **SpruceID** — Focused on self-sovereign identity and DID/verifiable credentials. Complementary but different problem space — SpruceID handles identity claims, Evidence Commons handles evidence claims with multi-party verification.

- **EAS SDK** — Provides the core attestation primitive that Evidence Commons builds upon. The EAS SDK handles single attestation creation, but offers no offline queuing, no schema-level impact standard, no multi-stage verification workflow, and no authentication bridge. Evidence Commons is the application-level toolkit built on top of EAS.

- **Silvi, GainForest, Regen Network** — Domain-specific impact platforms (tree monitoring, satellite-based verification, Cosmos-based ecological credits respectively). Each solves their specific use case but none produce reusable libraries that other projects can adopt. Green Goods is the only project extracting its infrastructure as open commons.

The key differentiator: the individual pieces exist (IndexedDB, EAS, WebAuthn, Service Workers) but nobody has published the integration layer — the reliable, tested pipeline that connects them into an offline-to-attestation flow for low-connectivity environments. Green Goods has already built and deployed this integration in production across four countries. Evidence Commons makes it available to everyone.

### Technical Challenges

_What are significant technical challenges you expect to solve during the project, if any?_

1. **IndexedDB cross-browser reliability** — Safari aggressively evicts IndexedDB storage under memory pressure. The queue library must degrade gracefully when storage is reclaimed mid-operation, and the file serialization layer (which currently handles iOS Safari DOMException edge cases in production) needs thorough cross-browser testing as a standalone library. We will implement storage quota detection and progressive fallback strategies.

2. **Background Sync API fragmentation** — The W3C Background Sync API is not universally supported (notably absent in Safari/iOS). The library needs robust fallback strategies — visibility change listeners, periodic polling with intelligent backoff, and manual sync triggers — that provide equivalent reliability across all browsers without requiring the developer to handle platform differences.

3. **Passkey portability and account recovery** — WebAuthn credentials are device-bound by design. The authentication bridge needs a clear account recovery story for when users lose their devices, without reintroducing the custodial dependencies that passkeys are designed to eliminate. We will explore social recovery and multi-device credential synchronization approaches.

4. **Attestation schema versioning** — Impact attestation schemas will evolve as the standard matures. The specification needs a forward-compatible versioning strategy that does not break existing on-chain attestations or require retroactive migration. We will design the schema format to support additive changes while maintaining backward compatibility.

5. **Abstraction without regression** — Extracting tightly-coupled production code into generic libraries risks introducing regressions in the working Green Goods application. The extraction must be done incrementally, with Green Goods itself as the first consumer of the new packages, ensuring the libraries remain battle-tested throughout development.
