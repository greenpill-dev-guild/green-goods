## Critical Findings

### C1. WithdrawModal uses "shares" input instead of "assets" (STILL OPEN)
- **File**: `packages/admin/src/components/Vault/WithdrawModal.tsx`
- **Lines**: 43, 73-83 (line 79 specifically for `parseUnits`)
- **Issue**: Users input share amounts directly, but ERC-4626 vault withdrawals should use asset amounts. Users don't know their share balance; they know how much underlying asset they want to withdraw. The `parseUnits(sharesInput, 18)` at line 79 hardcodes 18 decimals for shares regardless of actual vault share decimals.
- **Impact**: UX confusion, potential loss of funds if share/asset ratio is not 1:1 (which it won't be after yield accrues).
- **Recommendation**: Change input to accept asset amounts, use `previewWithdraw(assets)` to compute shares needed.

### C2. CreateGarden infinite re-render loop (STILL OPEN)
- **File**: `packages/admin/src/views/Gardens/CreateGarden.tsx:157-159`
- **Issue**: `useEffect(() => { resetValidationForm(form); }, [form, resetValidationForm])` -- `form` from `useShallow` on Zustand state may produce a new reference each render, causing RHF `reset()` to trigger re-render, which triggers the effect again.
- **Impact**: Potential browser freeze on the garden creation page.
- **Recommendation**: Stabilize `form` reference with deep comparison or move to a ref-based pattern.

- **File**: `packages/shared/src/hooks/hypercerts/hypercert-contracts.ts:43-51`
- **Issue**: Chain-specific contract addresses hardcoded as a fallback map. If Hypercerts team redeploys, this becomes stale silently.
