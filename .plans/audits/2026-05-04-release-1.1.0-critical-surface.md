## Findings

### 1. [HIGH] CEI violation in `GreenWill._issueBadge`: external `grantKeys` call before `_badgeRecords` write
- **File:** `packages/contracts/src/registries/GreenWill.sol:248-272`
- **Why it matters:** contract has no `ReentrancyGuard`. The duplicate-claim check (`if (_badgeRecords[badgeId][account].issued)` at line 168/221) reads state that hasn't been updated yet — a malicious-or-buggy `unlockLock` could re-enter `claimBadge` and mint multiple Unlock keys for the same badge. Owner controls lock configuration so the trust assumption mitigates blast radius, but the pattern is wrong on a funds-adjacent contract.
- **Proposed fix:** write `_badgeRecords[badgeId][account]` first (with placeholder `unlockTokenId = 0`), call `_grantUnlockKey`, then patch `unlockTokenId` — or add `nonReentrant` to `claimBadge` / `issueByAuthorizedIssuer` / `batchIssueEligible`.
- **Status:** DEFERRED to v1.1.1 hotfix tracked in [#513](https://github.com/greenpill-dev-guild/green-goods/issues/513) (2026-05-05). Trust mitigation (owner controls `unlockLock`; current canonical Unlock contracts on Arbitrum are non-adversarial) lets v1.1.0 ship with the finding documented. Issue captures the reorder-writes patch, 5 Forge re-entry tests, and the UUPS upgrade plan.

### 2. [HIGH] `useVaultWithdraw` ships with `maxLoss = 100%` and no caller-overridable slippage
- **File:** `packages/shared/src/hooks/vault/useVaultWithdraw.ts:65,80`
- **Constant:** `VAULT_MAX_BPS = 10000n` from `packages/shared/src/utils/blockchain/vaults.ts:10` (comment self-describes as "permissive")
- **Why it matters:** every withdrawal accepts arbitrary loss; user has no `minAssets` knob. Under price-oracle distress or vault degradation a user can be silently haircut to zero. `WithdrawParams` already accepts `receiver`/`owner` but no slippage field — the protection just doesn't exist.
- **Proposed fix:** add `maxLossBps?: bigint` (default 100 = 1%) or `minAssets?: bigint` to `WithdrawParams`; thread through to the `withdraw()` call instead of `VAULT_MAX_BPS`.
- **Status:** ADDRESSED in commit `e22c270e` (2026-05-05). `WithdrawParams.maxLossBps?: bigint` added, default 100n (1%), threaded through both `maxWithdraw` precheck and `withdraw` call. Two new tests in `useVaultOperations.test.ts` lock in default + override behavior. **Behavior change**: callers near the prior 100% tolerance threshold may now hit "exceeds available balance" — verified via TSC + lint + 2988/2989 shared tests + workspace build (no UI smoke this turn for `WithdrawModal.tsx` since admin's vitest config excludes it; consumer `MyDepositRow.tsx` has no test).

### 3. [HIGH] Yield routing semantics changed: `YieldStranded` events removed; missing-treasury now silently sweeps to garden address
- **File:** `packages/contracts/src/resolvers/Yield.sol:594-602, 800-829`
- **Why it matters:** old behavior emitted `YieldStranded(garden, asset, amount, source)` so off-chain monitoring could alert. New behavior treats the garden address itself as the treasury fallback when none is configured. Indexers/dashboards subscribed to `YieldStranded` are now blind to misconfigured gardens, and yield silently flows to whichever account controls the garden TBA.
- **Proposed fix:** confirm with stakeholders that the new semantics are intentional; remove or repoint any monitoring keyed on `YieldStranded`; document the change in operator-facing docs.

### 4. [HIGH] `setGardenTreasury` permission expanded from `onlyOwner` to garden operators
- **File:** `packages/contracts/src/resolvers/Yield.sol:391-399`
- **Why it matters:** a garden operator can now point yield treasury to any address. Combined with finding #3's silent fallback, an operator who removes their treasury and controls the garden TBA captures the next yield split. Test `test_setGardenTreasury_operatorCanSetTreasury` confirms the behavior is intentional, but the trust model needs documentation.
- **Proposed fix:** explicit human approval — accept the trust model and document it, or add a timelock / council-veto path.

### 5. [HIGH] Five new critical-surface vault mutation hooks land without paired unit tests
- **Files:**
  - `packages/shared/src/hooks/vault/useVaultDeposit.ts` (341 LOC, NEW)
  - `packages/shared/src/hooks/vault/useVaultWithdraw.ts` (120 LOC, NEW)
  - `packages/shared/src/hooks/vault/useHarvest.ts` (80 LOC, NEW)
  - `packages/shared/src/hooks/vault/useEmergencyPause.ts` (79 LOC, NEW)
  - `packages/shared/src/hooks/vault/useEnableAutoAllocate.ts` (90 LOC, NEW)
- **Coverage:** only `useVaultOperations.test.ts` references one of them; the new mutation surface is untested.
- **Why it matters:** per CLAUDE.md Criticality Matrix, vault hooks are `critical`; this is the funds path. `useVaultDeposit` alone has slippage, allowance-reset, USDT-style approval, and 5-way diagnostic branching — all untested. The next refactor regresses silently.
- **Proposed fix:** unit tests for at least `useVaultDeposit` (slippage, allowance flows, vault-shutdown/limit branches) and `useVaultWithdraw` (after #2's slippage param lands).
- **Status:** OVERSTATED. Re-investigation 2026-05-05 found `useVaultOperations.test.ts` already covers all 5 hooks via the `useVaultOperations.ts` re-export facade — 17 cases covering two-step deposit, allowance-sufficient skip, post-approval slippage detection, withdraw single-step / at-limit / above-limit reject, harvest, emergencyPause, enableAutoAllocate (incl. non-owner reject), passkey routing, error-handler integration, VaultDepositStageError diagnostics, and inline error mode. Commit `e22c270e` adds 2 more for #2's slippage default + override (now 19 total). Genuine remaining gaps that are narrower than the original framing: USDT-style allowance reset path (`allowance > 0` → zero → reapprove), `isRecoverableAllowanceReadError` recovery, and individual diagnostic branches (`vaultShutdown` / `depositLimitZero` / `depositLimitReached` per-case rather than aggregate). These are v1.1.1 scope, not release-blocking.

### 6. [MEDIUM] Silent `catch {}` swallows wallet disconnect errors in Auth provider
- **File:** `packages/shared/src/providers/Auth.tsx:339-343`
- **Why it matters:** violates `.claude/rules/typescript.md` Rule 4 (Error Handling Consistency). Wallet-adapter disconnect failures during sign-out are now invisible.
- **Proposed fix:** `logger.debug("[AuthProvider] disconnect failed", { error })` in the catch.

### 7. [MEDIUM] Silent `.catch(() => {})` on service-worker `clearAllCaches()` during sign-out
- **File:** `packages/shared/src/providers/Auth.tsx:456`
- **Why it matters:** failed cache clearing on sign-out leaves stale data accessible across sessions — a privacy/leak concern. Same Rule 4 violation.
- **Proposed fix:** log at warn level instead of swallowing.

### 8. [MEDIUM] Silent error swallowing on `jobQueue.getStats` failure
- **File:** `packages/shared/src/providers/JobQueue.tsx:107-116`
- **Why it matters:** stats fetch failures (IndexedDB corruption, migration breakage) hide silently. Same Rule 4 violation.
- **Proposed fix:** log non-abort errors at warn level.

### 9. [MEDIUM] `createAllowlistAndRegister` no longer supports open allowlists (`merkleRoot = 0` reverts)
- **File:** `packages/contracts/src/modules/Hypercerts.sol:152`
- **Why it matters:** behavior change. Old `mintAndRegister` accepted zero merkleRoot for open mints; new function reverts on `InvalidMerkleRoot`. If any UI flow / migration script / docs example used open allowlists, they break silently.
- **Proposed fix:** grep the codebase for callers of the old signature; document the open-allowlist removal; verify admin Hypercert wizard always supplies a non-zero root.

### 10. [MEDIUM] Hypercerts API rename: `mintAndRegister` → `registerHypercert` + `createAllowlistAndRegister`; new function requires caller to predict `expectedHypercertId`
- **File:** `packages/contracts/src/modules/Hypercerts.sol:113-164`
- **Why it matters:** API breaking. The `expectedHypercertId` arg couples the caller to the minter's ID-allocation logic — caller must read the next ID from the minter and pass it in, with no concurrent-mint protection.
- **Proposed fix:** verify all callers (admin wizard, scripts, fork tests) supply the predicted ID correctly; document the prediction protocol.

### 11. [MEDIUM] `DevAuthProvider` `actions` object has three keys that aren't in `AuthActionsValue`
- **File:** `packages/shared/src/providers/DevAuthProvider.tsx:103-129`
- **Extras:** `signInWithPasskey`, `createPasskey`, `connectWallet`
- **Why it matters:** type drift; if `AuthActionsValue` ever tightens excess-property checking these break the build, and right now they're dead handlers that confuse the dev-auth contract.
- **Proposed fix:** remove the three extra keys.

### 12. [MEDIUM] `job-executors.ts` re-uploads to IPFS on retry — no client-side idempotency
- **File:** `packages/shared/src/modules/job-queue/job-executors.ts:54-73` (per the `encodeWorkData` comment "includes IPFS upload")
- **Why it matters:** if the EAS submit fails after IPFS upload succeeded, the next retry re-uploads identical content → duplicate pins, wasted bandwidth. Recent commit `feat(agent): idempotency + rate-limit + key-encryption hardening` (359f059e) hardened the agent boundary; the client queue has no parallel hardening.
- **Proposed fix:** cache the IPFS CID (or encoded attestation bytes) on the job record so retries reuse it.

### 13. [MEDIUM] Asymmetric client-side eligibility checks across vault hooks
- **Files:**
  - `useEnableAutoAllocate.ts:31-40` — checks owner client-side
  - `useEmergencyPause.ts`, `useHarvest.ts` — no pre-check
- **Why it matters:** same surface, inconsistent UX — one path gives a clear "only the OctantModule owner can…" error, the others bubble a contract revert as a generic toast. UX-only, not security (contract enforces).
- **Proposed fix:** add owner/role pre-checks in `useEmergencyPause` and `useHarvest`, or remove the pre-check from `useEnableAutoAllocate` and rely on contract reverts uniformly.

### 14. [MEDIUM] `Gardens.isReadyForGardens` requires `yieldResolver` set, but `setYieldResolver(address(0))` is explicitly allowed
- **File:** `packages/contracts/src/modules/Gardens.sol:333-340, 449-453`
- **Setter dev comment:** "address(0) is allowed to disable auto-wiring on networks where YieldResolver is not deployed yet"
- **Why it matters:** contradiction. On a network where `yieldResolver` is intentionally `address(0)`, the readiness gate returns false. Either the readiness check should tolerate zero, or the setter should disallow it.
- **Proposed fix:** pick one — relax `isReadyForGardens` to allow `yieldResolver == address(0)`, or assert non-zero in `setYieldResolver`.

### 15. [MEDIUM] `as any` type-safety hole in `job-executors.ts` at the IPFS-upload boundary
- **File:** `packages/shared/src/modules/job-queue/job-executors.ts:44, 67`
- **Why it matters:** the draft type passed to `simulateWorkSubmission` and `encodeWorkData` is cast to `any`, so the IPFS-upload contract is unverified at compile time. Any future refactor of `WorkDraft` will silently diverge from what the executor actually sends.
- **Proposed fix:** define the simulate/encode draft type and use it; no `any`.

### 16. [LOW] Silent error swallow in `JobMaintenance.cleanupOrphanedSyncedJobs`
- **File:** `packages/shared/src/modules/job-queue/job-maintenance.ts:99-101`
- **Why it matters:** failed re-deletes get counted but not logged; persistent IndexedDB delete failures don't surface.
- **Proposed fix:** `logger.debug` (or warn at threshold) in the catch.

### 17. [LOW] Yield event rename (`YieldToCookieJar` → `YieldToTreasury`) not flagged for indexer migration
- **File:** `packages/contracts/src/resolvers/Yield.sol:91-100`
- **Why it matters:** the indexer and analytics dashboards keyed on the old event name silently stop indexing.
- **Proposed fix:** grep `packages/indexer/src/` and admin dashboards for `YieldToCookieJar`; migrate to `YieldToTreasury`.
